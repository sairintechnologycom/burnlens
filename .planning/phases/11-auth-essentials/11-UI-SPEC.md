# Phase 11 — UI Design Contract

> Visual and interaction contract for the Auth Essentials phase: password reset, email verification, transactional email templates, and the unverified-email banner. Generated by gsd-ui-researcher; pre-populated from `11-CONTEXT.md` (D-01–D-08 + Claude's Discretion) and the existing BurnLens design system in `frontend/src/app/globals.css` and `frontend/src/app/setup/page.tsx`.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual — pre-existing "BurnLens Option 2" system) |
| Preset | not applicable |
| Component library | none (handcrafted React + scoped `<style>` blocks per page) |
| Icon library | `lucide-react` (already installed; used in `BillingStatusBanner.tsx`) |
| Font | `var(--font-sans)` for UI body/headings; `var(--font-mono)` for labels, code-like inputs, footer chips (matches `setup/page.tsx`) |

**Source of truth:** `frontend/src/app/globals.css` (theme tokens, typography scale `--fs-10`…`--fs-32`, dark + light themes) and the inline `.sp-*` styles in `frontend/src/app/setup/page.tsx` (auth-page chrome). Phase 11 does NOT introduce a new design system. New pages reuse the `.sp-*` shell from `setup/page.tsx`.

**Why no shadcn:** The project ships zero `components.json`, zero shadcn primitives, zero Radix. All existing pages render handwritten markup with CSS variables. Introducing shadcn solely for two auth pages would fork the design system. Registry Safety Gate is therefore **not applicable**.

## Typography

> **Inheritance note:** This phase introduces **zero new typography roles**. The full BurnLens scale (5 sizes / 4 weights) is the existing locked production system already shipping across `/setup`, `/dashboard`, and `/settings` and defined in `frontend/src/app/globals.css` (`--fs-9` / `--fs-12` / `--fs-13` / `--fs-20` / `--fs-32`). Phase 11 **selects from this scale; it does not extend it**. The "Net-new" sub-table below lists only the role variants Phase 11 introduces. The "Reused (informational, locked v1.0)" sub-table is reproduced for executor convenience and is not redeclared by this phase.

### Net-new for Phase 11 (≤4 sizes, ≤2 weights)

These are the only typography roles Phase 11 declares as new. Both reuse existing scale values; the novelty is the role assignment, not the size/weight.

| Role | Size | Weight | Line Height | Family | Notes |
|------|------|--------|-------------|--------|-------|
| Verification banner body | 12px (`--fs-12`) | 400 | 1.4 | `--font-sans` | New role; size/weight already in production for footer + banner copy |
| `/verify-email` status heading variant | 32px (`--fs-32`) | 800 | 1.15 | `--font-sans` | New role on the auth-shell right-pane (success/error display heading); size/weight already shipping in `.sp-left h2` |

**Net-new declaration count:** 2 sizes (12, 32), 2 weights (400, 800). Within thresholds (≤4 sizes / ≤2 weights).

### Reused (informational, locked v1.0 — not redeclared by this phase)

## Color

Theme tokens already exist in `globals.css` for both dark (default) and light. Phase 11 introduces zero new colors; it consumes:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--bg)` `#0a0b0f` (dark) / `#f8f7f4` (light) | Page background on `/reset-password`, `/verify-email`, `/setup` extension |
| Secondary (30%) | `var(--bg2)` `#181b24` / `#e8e4dc` | Left-pane fill on auth pages, banner tinted backdrop via `color-mix` |
| Accent (10%) | `var(--cyan)` `#2dd4bf` / `#0d7a6e` | Primary CTA fill, focused-input border, success icon |
| Warning (banner) | `var(--amber)` `#f5a623` / `#b45309` | Email-verification banner border-left + link color (mirrors past-due banner) |
| Destructive | `var(--red)` `#f04060` / `#c0392b` | Inline form errors (`.sp-error`), `/verify-email` failure state |
| Success | `var(--green)` `#22c98a` / `#166534` | `/verify-email` success checkmark + heading underline accent |

**Accent reserved for** (explicit list — `var(--cyan)` only on these elements):

1. `.sp-btn-primary` background fill — "Send reset link", "Set new password", "Resend verification email", "Go to dashboard"
2. `.sp-input:focus` border color
3. `/verify-email` success state icon stroke (single 24px lucide `CheckCircle2`)
4. Inline "Forgot password?" link on `/setup` (same hover treatment as the existing footer links — color stays `var(--muted)`, hover transitions to `var(--cyan)`)
5. `.sp-left-feat-dot` bullets in the left-pane feature list (already present)

**Verification banner color rule (extends `BillingStatusBanner.tsx`):**

- Background: `color-mix(in srgb, var(--amber) 12%, var(--bg2))` — identical to past-due banner formula.
- Border-left: `3px solid var(--amber)`.
- Border-bottom: `1px solid var(--border)`.
- Icon: lucide `Mail` (14px) `color="var(--amber)"`.
- Body text: `var(--text)` at 12px, weight 400.
- Inline link: `var(--amber)` at weight 600, no underline.

**Stacking order when both banners are present:** Past-due (billing) banner renders first, verification banner renders directly below. Both share the 40px height and 1px bottom border. Total stacked banner block = 80px tall.

## Layout

Phase 11 introduces two new top-level routes plus extends one existing route:

### `/reset-password` (D-06)

- Reuses the `.sp-shell` two-pane structure from `setup/page.tsx`: left pane (brand + feature list) on viewport ≥ 960px, single right-pane stack below that breakpoint.
- Right pane vertical centering: `.sp-right { padding: 64px 48px; display: flex; flex-direction: column; justify-content: center; }`.
- Form max-width: 360px (matches `.sp-form` in setup).
- Vertical rhythm between fields: 16px (`.sp-field` margin-bottom).
- CTA button is full-width within the form.

### `/verify-email` (D-06)

- Same `.sp-shell` two-pane structure as `/reset-password`.
- Right-pane content is **state-driven** (loading / success / error — see States section). All states render inside the same right-pane container with no layout shift between transitions.
- Status heading + body + CTA stack vertically with 24px gap between heading and body, 32px gap before CTA.

### `/setup` (extension only — D-07)

- "Forgot password?" inline link added immediately below the password field on the existing `/setup` page.
- Visual: 13px sub-text, `var(--muted)` color, hover transitions to `var(--cyan)`. Matches existing footer-link treatment.
- Position: right-aligned under the password input, 8px above the next form element.

### Dashboard banner stack

- `BillingStatusBanner` is mounted in `Shell.tsx` directly below `<Topbar />`. Phase 11 mounts the new `EmailVerificationBanner` in the same slot, **after** the past-due banner so it stacks below when both are visible.
- When only verification banner is visible: total banner block = 40px tall, pushes content down by 40px.
- When both banners are visible: total banner block = 80px tall, pushes content down by 80px. No horizontal layout change.

## States

Each new page must implement the following states explicitly. No layout shift between states within the same page.

### `/reset-password` (request flow inline on `/setup`)

| State | Trigger | Visible Elements |
|-------|---------|------------------|
| Idle | Page load | Email input + "Send reset link" CTA |
| Submitting | After CTA click | CTA shows "Sending…", input disabled, no spinner overlay |
| Success | 200 response | Replaces form with success message text (AUTH-01 neutral message) |
| Rate-limited | 429 response | Inline error below input, form remains editable |

### `/reset-password/confirm` (new-password page)

| State | Trigger | Visible Elements |
|-------|---------|------------------|
| Idle | Page load with valid token | Both password fields + CTA |
| Submitting | After CTA click | CTA shows "Saving…", inputs disabled |
| Success | 200 response | Replaces form with success heading + body + CTA to `/setup` |
| Validation error (mismatch / length) | Client-side | Inline `.sp-error` below relevant field, CTA remains active |
| Token error (invalid / expired) | 400/404 from confirm endpoint | Replaces form with error heading + body + link back to `/setup` |

### `/verify-email`

| State | Trigger | Visible Elements |
|-------|---------|------------------|
| Loading | Page mount, before backend response | Centered "Verifying your email…" body, no heading, no CTA |
| Success | 200 from `/auth/verify-email?token=…` | 32px success heading, body, CTA to dashboard, lucide `CheckCircle2` icon at 24px in `var(--green)` above heading |
| Error | 400/404 (invalid or expired token) | 32px error heading (color `var(--red)`), body, CTA to dashboard, lucide `XCircle` icon at 24px in `var(--red)` above heading |

### Email verification banner (in dashboard)

| State | Trigger | Visible Elements |
|-------|---------|------------------|
| Hidden | `session.emailVerified === true` OR session loading | Nothing rendered |
| Visible | `session.emailVerified === false` | Full banner with `Mail` icon + body + resend link |
| Resend pending | After resend link click | Link text changes to "Sending…", disabled |
| Resend success | 200 response | Link replaced inline with confirmation copy for ~5s, then reverts to "resend verification email" |
| Resend rate-limited | 429 response | Inline rate-limit copy replaces link for ~5s |

### Transactional email templates

States are not interactive. Templates render once with inlined CSS. Each template must include:

- Visible plain-text heading mirroring the subject
- BurnLens wordmark (text only — no logo image, matches existing usage_*_percent templates)
- Body paragraph(s)
- Action CTA where applicable (verify, reset, dashboard, receipt detail link)
- Footer with `noreply@burnlens.app` sender attribution and unsubscribe-not-applicable note for transactional class

## Accessibility

| Concern | Requirement |
|---------|-------------|
| Color contrast | All text combinations must pass WCAG AA (4.5:1 for body, 3:1 for ≥18px). Existing `globals.css` tokens have been audited in v1.0 and pass. New banner amber-on-tinted-bg combination is identical to past-due banner — already passing. |
| Focus visibility | All interactive elements use the existing `.sp-input:focus` border treatment (2px `var(--cyan)` ring). Buttons inherit browser default focus ring; no override that removes it. |
| Form labels | All inputs have explicit `<label htmlFor>` matching `id` (matches `setup/page.tsx` pattern). Placeholder text is supplemental, never the only label. |
| Error association | Inline errors use `aria-describedby` linking input to its `.sp-error` element. |
| Banner semantics | Verification banner uses `role="status"` + `aria-live="polite"` (matches past-due banner). Resend link uses `<button type="button">` not `<a>` since it triggers a backend action. |
| Loading state | `/verify-email` loading state body has `aria-live="polite"` so transition to success/error is announced. |
| Keyboard navigation | Tab order: form fields top-to-bottom, then CTA. No custom tab traps. |
| Email templates | Plain-text alternative provided for each HTML template (existing `email.py` pattern requires both). |

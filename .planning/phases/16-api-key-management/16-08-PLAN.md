<objective>
Close CR-02: `burnlens_cloud/auth.py::resend_verification` unconditionally calls `decrypt_pii(rows[0]['email_encrypted'])`. If the blob is NULL (rotated PII master key, partial Phase 1c backfill, dev-row, etc.) or `decrypt_pii` raises, the handler 500s. This violates D-14 (always-200 enumeration-safe) and CLAUDE.md fail-open posture — a 500 leaks "this user_id exists in degraded state" and is distinguishable from the verified-or-missing 200 path.

Purpose: Make resend-verification fail-open: any path that can't recover the email returns the same 200 envelope without sending an email, so callers cannot distinguish "no such user", "already verified", "blob NULL", or "decrypt failed" from each other.
Output: Hardened `resend_verification` handler + two regression tests (NULL blob, decrypt raises) in `tests/test_phase16_auth08_resend.py`.
</objective>

    return {"message": "If applicable, a verification email has been sent."}

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→cloud API | Unauthenticated POST attempts hit `verify_token` first — already 401. The threat surface is **post-auth**: an attacker with a stolen valid JWT (or a user accessing their own degraded account) could distinguish account states via response code if the handler is unguarded. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-08-01 | Information Disclosure | `burnlens_cloud/auth.py::resend_verification` | mitigate | An unguarded `decrypt_pii(NULL)` or `decrypt_pii(corrupt_blob)` 500 is a side-channel that lets a caller distinguish "valid JWT for a degraded-state user" from the always-200 verified/missing/unverified paths. Mitigation: NULL guard + bare-Exception fail-open returning the same 200 envelope. Verified by `test_resend_verification_handles_null_email_encrypted` and `test_resend_verification_handles_decrypt_error`. HIGH severity — D-14 explicitly designates this endpoint as enumeration-safe. |
| T-16-08-02 | Denial of Service | `decrypt_pii` failure storm | accept | If the PII master key is rotated incorrectly and every user becomes "degraded", every resend attempt now logs at WARNING and returns 200 — no cascading 500s, no panic. Log volume is bounded by user click-throughs on the verify banner; not a useful DoS vector. |
| T-16-08-03 | Repudiation | Missing audit trail on degraded-state attempts | accept | The two new `logger.warning(...)` calls record `user_id` + reason — sufficient post-hoc audit for a key-rotation incident. No structured event emission needed. |

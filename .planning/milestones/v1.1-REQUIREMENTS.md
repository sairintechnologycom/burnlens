## Milestone v1.1: Billing & Quota

**Goal:** Surface the user's tier, let them manage billing in-app, and enforce plan limits so free/Cloud/Teams users can't exceed what they paid for.

**Started:** 2026-04-18
**Completed:** 2026-04-30

---

## v1.1 Requirements

### Billing Panel (BILL)

- [x] **BILL-01** — User can view current plan name, price, and subscription status (active / trialing / past_due / canceled) from Settings → Billing
  *Validated Phase 7: GET /billing/summary + Settings Billing card (plan+price+status pill)*
- [x] **BILL-02** — User can see next billing date and, if trialing, the trial expiry date
  *Validated Phase 7: current_period_ends_at + trial_ends_at in BillingSummary*
- [x] **BILL-03** — User can launch a Paddle checkout to upgrade or downgrade their plan without leaving the app
  *Validated Phase 8: POST /billing/change-plan + PlanPickerModal + usePaddleCheckout hook*
- [x] **BILL-04** — User can view invoice history with amount, date, status, and a download link (via Paddle customer portal)
  *Validated Phase 8: GET /billing/invoices + InvoicesCard with PDF download*
- [x] **BILL-05** — User can cancel their subscription self-serve (cancel-at-period-end) and see the effective end date
  *Validated Phase 8: POST /billing/cancel + CancelSubscriptionModal + cancel_at_period_end flag*
- [x] **BILL-06** — User can reactivate a canceled-but-not-ended subscription before the period expires
  *Validated Phase 8: POST /billing/reactivate (400 if period already ended)*

### Plan Definitions (PLAN)

- [x] **PLAN-01** — Plan limits (monthly request cap, seat count, retention days, API key count, gated features) are persisted in a `plan_limits` Postgres table as the source of truth
  *Validated Phase 6: plan_limits table with 9 columns, partial index on paddle_price_id*
- [x] **PLAN-02** — Workspace can carry per-workspace limit overrides that supersede its plan's defaults (for enterprise exceptions)
  *Validated Phase 6: workspaces.limit_overrides JSONB nullable column, COALESCE merge in resolve_limits()*
- [x] **PLAN-03** — Three seeded plans exist at migration time: Free (local only), Cloud ($29/mo), Teams ($99/mo) — values confirmed against the live Paddle products
  *Validated Phase 6: 3 seed rows with live Paddle price IDs*
- [x] **PLAN-04** — A single resolver function returns the effective limits for a given workspace (workspace override > plan default)
  *Validated Phase 6: resolve_limits(ws_id UUID) SQL function + burnlens_cloud/plans.py async wrapper*

### Quota Enforcement (QUOTA)

- [x] **QUOTA-01** — Server tracks monthly request count per workspace, reset on each billing cycle anniversary
  *Validated Phase 9: workspace_usage_cycles table + UPSERT in ingest.py + Paddle webhook cycle rollover*
- [x] **QUOTA-02** — Workspace receives an email when monthly usage crosses 80% and again at 100% of plan quota
  *Validated Phase 9: send_usage_warning_email + 80%/100% HTML templates + ingest.py enqueue*
- [x] **QUOTA-03** — `POST /v1/ingest` records usage but does NOT hard-reject over-quota workspaces in v1.1 (soft enforcement)
  *Validated Phase 9: counter UPSERT in ingest.py; 429 removed; hard rejection deferred to v1.2*
- [x] **QUOTA-04** — Team invite endpoint returns 402 Payment Required when inviting a member would exceed the plan's seat limit
  *Validated Phase 9: team_api 422→402 + get_seat_limit via resolve_limits*
- [x] **QUOTA-05** — A scheduled job prunes stored events older than the workspace's plan retention window
  *Validated Phase 9: retention-prune asyncio loop in main.py lifespan wiring*

### Feature Gating (GATE)

- [x] **GATE-01** — Free-tier workspaces can sync to cloud but cannot access Teams or Customers views (UI locked with upgrade CTA)
  *Validated Phase 10: LockedPanel with frosted-glass teaser + Paddle checkout CTA on /teams and /customers*
- [x] **GATE-02** — Cloud-tier workspaces see the Teams tab as locked with "Upgrade to Teams" CTA; Teams-tier sees it unlocked
  *Validated Phase 10: planSatisfies() + LOCKED_NAV map + Sidebar.lockedForPlan affordance*
- [x] **GATE-03** — Customer attribution views require Teams plan; Cloud and Free see upgrade CTA
  *Validated Phase 10: /customers migrated to LockedPanel; UpgradePrompt.tsx deleted*
- [x] **GATE-04** — API key creation endpoint enforces per-plan key count (Free: 1, Cloud: 5, Teams: unlimited)
  *Validated Phase 9: /api-keys POST returns 402 when api_keys count ≥ plan limit*
- [x] **GATE-05** — Backend middleware verifies plan entitlement on every gated API call; 402 returned if required tier not met
  *Validated Phase 9: require_feature(name) FastAPI dependency factory on gated routes*

### Usage Visibility (METER)

- [x] **METER-01** — Every dashboard page displays a usage meter in the sidebar: current month requests / plan limit, with a progress bar
  *Validated Phase 10: UsageMeter in Sidebar footer consuming BillingContext.usage*
- [x] **METER-02** — Usage meter color transitions: green (<80%), amber (80–100%), red (>100%)
  *Validated Phase 10: threshold coloring in UsageMeter; ARIA progressbar + polite live-region*
- [x] **METER-03** — Clicking the usage meter deep-links to Settings → Billing → Usage section with a daily breakdown
  *Validated Phase 10: UsageCard with #usage anchor + VerticalBar daily chart + GET /billing/usage/daily*

### Paddle Integration (PDL)

- [x] **PDL-01** — Webhook handler processes `subscription.created`, `subscription.updated`, `subscription.canceled` and updates the workspace plan state
  *Validated Phase 7: handlers for subscription.activated/updated/canceled/paused; ON CONFLICT dedup*
- [x] **PDL-02** — Webhook handler processes `transaction.completed` and `transaction.payment_failed` to drive `past_due` and `active` transitions
  *Validated Phase 7 (partial): transaction.payment_failed sets past_due; transaction.completed recovery via subscription.updated(active) — documented design choice in 07-02 SUMMARY*
- [x] **PDL-03** — Webhook signatures are verified via the Paddle webhook secret before any state mutation
  *Validated Phase 7: signature-first 401 gate before any DB write*
- [x] **PDL-04** — Plan badge in the Topbar reflects subscription state accurately within 60 seconds of a Paddle lifecycle event
  *Validated Phase 7: Topbar pill rewired to useBilling().billing.plan; BillingContext 60s polling*

## Notes on Changed Requirements

- **PDL-02** adjusted: `transaction.completed` explicit routing deferred; recovery via `subscription.updated(status=active)` confirmed as sufficient. Documented design choice.
- **QUOTA-03** scoped: hard 429 rejection explicitly deferred to v1.2 — soft enforcement is the v1.1 contract.
- **GATE-04/05** scope clarified during Phase 9: entitlement middleware applies at the route level, not just at the ingest boundary.

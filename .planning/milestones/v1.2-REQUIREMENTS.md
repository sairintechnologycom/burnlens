# Requirements Archive — v1.2 Account Security & Notifications

**Milestone:** v1.2
**Archived:** 2026-05-07
**Status:** ✅ All requirements shipped

---

## Auth — Password Reset

- [x] **AUTH-01**: User can request a password reset via email (endpoint always returns 200 — no user enumeration)
- [x] **AUTH-02**: User can set a new password via a time-limited, single-use reset token link
- [x] **AUTH-03**: Password reset request endpoint is rate-limited (reuses existing `rate_limit.py`)

## Auth — Email Verification

- [x] **AUTH-04**: New signups automatically trigger an email verification email
- [x] **AUTH-05**: User can confirm their email ownership via a verification token link
- [x] **AUTH-06**: Unverified users see a persistent dashboard banner prompting verification (soft-gate — no hard block in MVP)
- [x] **AUTH-07**: Existing users (pre-v1.2, `email_verified_at = NULL`) are grandfathered as verified automatically — no action required

## Email — Transactional System

- [x] **EMAIL-01**: User receives a welcome email on successful signup
- [x] **EMAIL-02**: User receives a password-changed confirmation email after completing a reset
- [x] **EMAIL-03**: User receives a payment receipt email after a successful Paddle payment (wired into existing `payment_succeeded` webhook handler)
- [x] **EMAIL-04**: Transactional system supports adding new templates via a typed template registry (extends existing `burnlens_cloud/email.py` beyond invitation-only)

## Alert — Schema & Seeding

- [x] **ALERT-01**: Cloud plan workspaces have default budget alert rules auto-seeded at 80% and 100% of their plan's monthly allowance
- [x] **ALERT-02**: Alert events table records a deduplication + audit log of every alert that fired (what rule, when, to whom)

## Alert — Cron & Dispatch

- [x] **ALERT-03**: An hourly Railway cron job evaluates all active alert rules against per-workspace spend
- [x] **ALERT-04**: Triggered alerts are deduplicated — the same rule does not re-notify within a 24-hour window
- [x] **ALERT-05**: Alert evaluation failures are logged but never block the cron (fail-open)
- [x] **ALERT-06**: Org owner receives an email notification (via SendGrid) when a budget threshold is crossed
- [x] **ALERT-07**: Org owner can optionally configure a per-workspace Slack webhook to receive threshold notifications

## Alert — Management UI

- [x] **ALERT-08**: Org owner can view all alert rules for their workspace on the `/alerts` page
- [x] **ALERT-09**: Org owner can enable/disable a rule, edit its threshold, and manage notification email recipients

## Routing — Budget-Aware Model Downgrade

- [x] **ROUTE-01**: When a workspace's remaining budget drops below a configurable threshold, the proxy silently routes to the cheapest available model tier — caller receives a valid response with no error
- [x] **ROUTE-02**: Routing logic fails open — any exception (DB error, config error, missing map entry) results in the original model being used; proxy never blocks due to routing
- [x] **ROUTE-03**: `requests` table records `routed_model` and `downgrade_reason` for every proxied request; downgraded rows have non-NULL `downgrade_reason`
- [x] **ROUTE-04**: Dashboard "Downgrades Today" stat card shows correct counts and estimated savings pulled from `/api/routing-stats`
- [x] **ROUTE-05**: `burnlens routing` CLI command shows a table of all downgrade events with timestamp, original model, routed model, reason, and budget remaining
- [x] **ROUTE-06**: Routing is controlled by `burnlens.yaml` config (`routing.budget_downgrade`, `routing.downgrade_threshold_pct`, `routing.downgrade_threshold_usd`) — disabled by default, zero code change for the user
- [x] **ROUTE-07**: Test suite covers 12 test cases for `decide_route()`: budget healthy pass-through, pct threshold trigger, USD threshold trigger, both thresholds, fail-open on exceptions, team spend cache, and provider map lookups

const fetchRules = useCallback(async () => {
  if (!session) return;
  setLoading(true);
  setError("");
  try {
    const data = await apiFetch("/api/v1/alert-rules", session.token);
    setRules(Array.isArray(data) ? data : []);
  } catch (err: any) {
    if (err instanceof AuthError) logout();
    else setError(err.message);
  } finally {
    setLoading(false);
  }
}, [session, logout]);
useEffect(() => { fetchRules(); }, [fetchRules]);

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → GET /api/v1/alert-rules | Token from session cookie (managed by useAuth); no attacker-controlled workspace_id |
| browser → PATCH /api/v1/alert-rules/{rule_id} | rule_id from fetched data, not user-typed; token from session |
| email chip input → editEmails array | User-typed email string; validated client-side for format before adding |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-02-01 | Information Disclosure | alerts/page.tsx | accept | Page only renders what the API returns; has_slack bool (not URL) already enforced at API layer (Plan 01). Frontend has no additional exposure. |
| T-13-02-02 | Tampering | PATCH via handleToggle | mitigate | Optimistic revert on any non-2xx response; AuthError forces logout; no stale state left on failure |
| T-13-02-03 | Tampering | PATCH via handleSave (extra_emails) | mitigate | Client-side "@" + "." validation before chip is added; invalid entries rejected before PATCH is sent |
| T-13-02-04 | Tampering | Edit modal — Enter key on chip input | mitigate | e.preventDefault() on Enter in onKeyDown handler prevents accidental form submission (per UI-SPEC Interaction Contract §3 and RESEARCH Pitfall 4) |
| T-13-02-05 | Elevation of Privilege | Toggle / Edit UI | mitigate | session.role === "owner" gate applied at render — viewer sees no toggle button and no Actions column; DOM-only guard, backend enforces 403 independently |
| T-13-02-06 | Denial of Service | Extra emails array length | accept | No client-side cap on chip count; backend stores in TEXT[]; no server resource risk. Low-value target — notification list only. |
</threat_model>

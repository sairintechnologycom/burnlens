# v1.3 — Cloud-side Per-API-Key Spend Governance (scoping doc)

**Status:** Queued. Activate with `/gsd-new-milestone` after v1.2 (Auth Essentials + Cloud Alerting) completes.
**Problem:** OSS CODE-2 ships per-API-key daily hard caps that work brilliantly on a developer's laptop, but Teams-tier customers have no way to set or enforce caps **across an organisation**. An admin can't set "$50/day on Alice's prod-openai key" from the cloud dashboard, and a leaked key on a teammate's machine that doesn't run the local proxy is invisible until month-end.
**Non-goals:** Replacing local enforcement (the local kill-switch stays the fastest, cheapest line of defence). Cross-provider rate limits. Real-time per-request blocking via cloud — proxy fan-out latency makes this unworkable for a Teams tier; we lean on **near-real-time visibility + after-the-fact freeze** instead.

**Depends on:**
- v1.1 Phase 9 (workspace-scoped `api_keys` table on cloud — currently next-up)
- v1.2 Phase B (cloud alerting cron infra — needed for cap-breach notifications)
- OSS P-1 (cloud sync of `tag_key_label` column — must precede this milestone)

---

## Phase A — Cloud Key Registry & Visibility

**Goal:** A Teams-tier admin can see every registered API key across their workspace, who it belongs to, today's spend per key, and the local cap each member configured. Read-only — no enforcement yet.

### Deliverables

1. **Cloud `api_keys` schema extension**
   - Existing v1.1 Phase 9 `api_keys` table gains: `key_label`, `provider`, `daily_cap_usd`, `cap_source` (enum: `local` / `cloud_override`), `last_synced_at`, `synced_by_user_id`
   - Local proxy sync payload extends `/v1/ingest` to include the user's registered keys (label + cap only — the SHA-256 hash stays local; cloud receives a workspace-scoped opaque ID)
   - Migration: backfill `cap_source = 'local'` for any pre-existing rows

2. **Per-key spend rollup**
   - New `workspace_key_spend_daily` materialized view: `workspace_id`, `key_label`, `user_id`, `day`, `spend_usd`, `request_count`
   - Refreshed by the existing ingest pipeline (counter UPSERT on each batch)
   - Indexed on `(workspace_id, day DESC)` for the "today" panel and on `(workspace_id, key_label)` for drill-down

3. **Read API**
   - `GET /api/v1/keys` — list registered keys in the workspace (admin only)
   - `GET /api/v1/keys/{label}/spend?days=N` — historical roll-up for one key
   - `GET /api/v1/keys-today` — cloud equivalent of OSS `/api/keys-today` (admin sees all members; member sees own keys)

4. **Dashboard UI**
   - New page `/keys` — table of (member · key label · provider · today's spend · cap · status)
   - Badge column matches the OSS dashboard: green / amber (≥80%) / red (≥100%)
   - Click-through to per-key historical chart
   - RBAC: owners + admins see all; viewers see own keys only

### Touches
`burnlens_cloud/database.py` (migrations), `burnlens_cloud/keys.py` (new module), `burnlens_cloud/ingest.py` (key sync handler), `burnlens/cloud/sync.py` (OSS-side payload extension), `frontend/src/app/keys/page.tsx` (new), `frontend/src/lib/billing.ts` (RBAC helper extension).

### Risks / unknowns
- **Privacy posture:** today's contract is "no key material leaves the machine." Sending the *label* is fine; sending the SHA-256 *hash* would let cloud correlate keys across workspaces — pick one and document it in `PRIVACY.md`.
- **Sync cadence:** keys today, registered locally at 09:00, may not appear in the cloud dashboard until the next sync (60s default). Document the lag; do not promise real-time.
- **Disconnected proxies:** a member who hasn't run `burnlens start` in 30 days has stale data. Show a "last synced" column; flag rows older than 24h.

## Phase B — Cloud-Authored Cap Overrides (soft)

**Goal:** A Teams admin can set a cap on any member's key from the cloud dashboard. The local proxy honours the cloud cap on next sync. Soft enforcement — admin gets an alert if the member's local proxy is offline or out of sync.

### Deliverables

1. **Override write path**
   - `PATCH /api/v1/keys/{label}` accepts `daily_cap_usd` + reason string; writes `cap_source = 'cloud_override'`, `overridden_by_user_id`, `overridden_at`
   - Audit row in `audit_log` (existing v1.1 table)
   - Email notification to the key owner: "Admin set a $X/day cap on your `prod-openai` key"

2. **Local proxy pull**
   - OSS `burnlens/cloud/sync.py` extended with a *pull* leg (currently push-only) that fetches active overrides on each sync cycle
   - Local key store applies override with precedence: `cloud_override` > `local`, but never *below* a stricter local cap (admin can lower, user can lower further; user cannot raise above admin)
   - On conflict, local proxy logs once per day to terminal and dashboard "API keys today" panel surfaces a `cap source: cloud (admin)` badge

3. **Drift detection alert**
   - New rule type in `cloud_alerts` (extends v1.2 Phase B engine): `cap_drift` — fires when a member's `last_synced_at` is older than X hours AND they have an active cloud override
   - Default rule: 24h drift → email admin
   - Suppresses if the member's proxy reports "offline mode acknowledged"

### Touches
`burnlens_cloud/keys.py`, `burnlens_cloud/cloud_alerts/rules.py`, `burnlens/cloud/sync.py` (pull leg + conflict resolution), `burnlens/keys.py` (precedence logic), frontend `/keys` page (edit modal), `frontend/src/components/KeyCapEditor.tsx` (new).

### Risks / unknowns
- **Race condition:** member registers a new key locally → first sync may push it up before any cap is set, leaving a window. Mitigation: default cap from workspace policy (Phase C), so new keys inherit a sane ceiling immediately.
- **Offline proxy:** if a member's proxy is offline for 7 days and an admin sets a cap, the cap is unenforced for that window. Phase B does not solve this — Phase C does.

## Phase C — Cloud-Side Hard Enforcement (true Teams kill-switch)

**Goal:** Even if a member's local proxy is off, dead, or never installed, the cloud can still enforce a per-key daily cap on traffic that flows through `api.burnlens.app` as a transparent forward-proxy (opt-in for Teams customers).

### Deliverables

1. **Cloud forward-proxy mode**
   - New endpoint: `POST /v1/proxy/{provider}/{path:path}` — accepts the same SDK-shaped requests as the OSS proxy
   - Authenticates with `bl_live_xxx` workspace key (existing) + the upstream provider key in `Authorization`
   - Streams responses through (SSE-aware, same passthrough rules as OSS)
   - Logs cost on response, evaluates cap server-side, returns `429 {"error": "burnlens_daily_cap_exceeded"}` at 100%

2. **Customer setup**
   - New env var: `OPENAI_BASE_URL=https://api.burnlens.app/v1/proxy/openai/v1`
   - Replaces the local proxy URL — customer trades local privacy for cloud enforcement (explicit opt-in per workspace)
   - `/keys` page surfaces a "cloud enforcement" toggle per key; default OFF

3. **Cost-attribution merge**
   - Same `key_label` + `tag_*` columns as Phase A, but written directly by the cloud forward-proxy (no sync delay)
   - Dashboard auto-detects which path each request came in via (`source = local_sync | cloud_proxy`)

### Touches
New module: `burnlens_cloud/forward_proxy/` (handlers, streaming, enforcement). Heavy lift on Railway egress + connection pooling. Migration to add `source` column on `request_records`. Frontend toggle + setup-page documentation.

### Risks / unknowns
- **Latency budget:** OSS proxy's <20ms overhead is the headline number. A cloud forward-proxy adds ~50–150ms RTT depending on user location. Need to either (a) accept the latency and document it, or (b) deploy multi-region (Railway → Fly.io migration?).
- **Egress cost:** cloud-proxying every byte of every LLM response (think 50KB/streamed-response × millions/month) is materially more expensive than ingesting cost rollups. Pricing model needs to absorb this — likely a usage-based add-on on the Teams tier.
- **Provider TOS:** OpenAI / Anthropic generally allow proxying, but check current terms before launch — a hosted forward proxy is a different posture from a local one.
- **Failure mode:** if the cloud proxy is down, the customer's app is down. Need a documented failover (fall back to direct provider URL on 5xx? leave as opt-in feature for cap-tolerant customers only?).

---

## Decisions made
- **Order:** Phase A → Phase B → Phase C. Each phase is independently shippable; Teams tier launches with A+B; C is gated behind explicit opt-in for the subset of customers who want server-side enforcement.
- **Local stays primary.** Local enforcement is the headline differentiator (per `ROADMAP-OSS.md` line 24). Cloud is additive, not replacement.
- **No SHA-256 hashes leave the machine.** Phase A uses workspace-scoped opaque IDs to preserve the privacy contract.

## Open questions for kickoff
1. **Pricing tier for Phase C:** Teams-included, or new Enterprise add-on? Latency + egress cost says add-on; competitive landscape says include.
2. **Multi-region:** does Phase C ship single-region (US-East) and add EU later, or wait until multi-region is ready before launching?
3. **Override precedence on lowering:** if admin sets $50, user sets $20 locally — does the user's $20 win, or does the cloud override pin the floor at $50? (Current spec: user can always go *stricter*, never looser. Confirm.)
4. **Audit-log retention:** v1.1 audit log retention follows plan tier. Cloud cap overrides are compliance-relevant — do they get a permanent retention exemption, or follow the tier?
5. **Per-team vs per-key:** S-1 (OSS 0.3.0) ships per-team monthly budgets locally. Should Phase A surface team budgets in the same `/keys` view, or keep keys and teams as separate dashboard tabs?

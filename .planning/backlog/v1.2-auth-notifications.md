# v1.2 — Account Security & Notifications (scoping doc)

**Status:** Queued. Activate with `/gsd-new-milestone` after v1.1 completes.
**Problem:** Cloud signups today are trusted-on-first-submit — no email verification, no password recovery, and no server-side spend alerts. The local proxy has a full alert engine (`burnlens/alerts/*`), but cloud users who never run the proxy get zero notifications.
**Non-goals:** SSO, MFA, SMS, mobile push. Not in this milestone — planned for later.

---

## Phase A — Auth Essentials

**Goal:** A cloud user can reset a forgotten password and prove they own the email they signed up with. Transactional emails (welcome, password-changed, receipt) are wired into the signup and billing flows.

### Deliverables
1. **Password reset flow**
   - `password_reset_tokens` table: `token_hash`, `user_id`, `expires_at`, `used_at`
   - `POST /auth/password-reset/request` — accepts email, always returns 200 (no user enumeration), sends email if user exists
   - `POST /auth/password-reset/confirm` — accepts token + new password, invalidates token, updates `password_hash`
   - Frontend: `/forgot-password` and `/reset-password/[token]` pages
   - Rate-limit the request endpoint (existing `rate_limit.py`)

2. **Email verification**
   - `email_verification_tokens` table (same shape as reset tokens)
   - `email_verified_at` column on `users`
   - `POST /auth/verify-email/request` and `/confirm`
   - Signup triggers a verification email automatically
   - Frontend: `/verify-email/[token]` page + banner on dashboard when unverified
   - Decision: soft-gate (banner only) for MVP; hard-gate (block billing changes) deferred

3. **Transactional email system**
   - Generalize `burnlens_cloud/email.py` beyond invitation-only — template registry + typed sender
   - HTML templates: `welcome.html`, `password_reset.html`, `password_changed.html`, `verify_email.html`, `receipt.html`
   - Wire `welcome.html` into `POST /signup`
   - Wire `password_changed.html` into reset-confirm
   - Wire `receipt.html` into Paddle webhook (`payment_succeeded`) — reuses v1.1 webhook handler

### Touches
`burnlens_cloud/auth.py`, `burnlens_cloud/email.py`, `burnlens_cloud/database.py` (migrations), `burnlens_cloud/emails/templates/`, `frontend/src/app/(auth)/*` routes.

### Risks / unknowns
- SendGrid deliverability for password-reset (time-sensitive) — may need dedicated IP or SPF/DKIM tuning
- Token secret storage — hash tokens at rest, never store raw
- Existing users (pre-v1.2) have `email_verified_at = NULL` — decide: grandfather in or force re-verify?

## Phase B — Cloud Alerting

**Goal:** An org owner receives an email when their workspace crosses a spend threshold, whether or not the local proxy is running. Mirrors the local `burnlens/alerts/engine.py` budget behavior at the cloud level.

### Deliverables
1. **Alert rules schema**
   - `alert_rules` table: `workspace_id`, `rule_type` (budget/spike/shadow), `threshold`, `window`, `notify_emails`, `enabled`
   - `alert_events` table: dedup + audit log of what fired when
   - Default rules auto-seeded per plan (Cloud = monthly budget alert at 80%/100% of plan allowance)

2. **Evaluation cron**
   - Railway cron job (hourly) that runs `cloud_alerts.engine.evaluate_all()`
   - Queries `request_records` per workspace, compares against active rules
   - Dedup window prevents re-firing within 24h
   - Fail-open: errors logged, never block the cron

3. **Notification dispatch**
   - Email via existing SendGrid client (`budget_alert.html` template)
   - Slack webhook per workspace (optional, stored in `workspace_settings`)
   - No SMS, no PagerDuty — future work

4. **Management UI**
   - `/alerts` page lists rules per workspace
   - Enable/disable, edit threshold, add/remove notification emails
   - Dashboard API: `GET/POST/PATCH/DELETE /api/v1/alert-rules`

### Touches
New module: `burnlens_cloud/cloud_alerts/` (engine, dispatch, rules). Schema migrations. Railway cron config. Frontend `/alerts` page (currently stubbed).

### Risks / unknowns
- Cron infra on Railway — validate existing setup or stand up a scheduler service
- Double-notification if user runs both local proxy alerts AND cloud alerts — document the overlap, or suppress cloud alerts when local alert engine was active for that window
- Spike-detection tuning (z-score? percentage jump?) — deferred; MVP is fixed budget thresholds only

---

## Decisions made
- **Order:** Phase A (Auth Essentials) first, then Phase B (Cloud Alerting). Rationale: no point emailing spend alerts to unverified addresses.

## Open questions for kickoff
1. **Transactional provider:** SendGrid vs. Resend vs. Postmark. Currently on SendGrid (invite emails work) — stay or switch?
2. **Existing users:** grandfather as verified, or force re-verify on next login?
3. **Railway cron:** is scheduler infra already wired, or does Phase B need a bootstrap sub-task to stand it up?
